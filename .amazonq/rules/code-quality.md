# Code Quality Rules

## Dart / Flutter

### State Management
- `HomeScreen` (and any future top-level screen) must not own business logic directly in `State` — extract to a `ChangeNotifier` or equivalent controller class when a `State` class exceeds ~150 lines or manages more than two independent concerns
- Controllers live in `lib/controllers/`; screens only call methods and read state from them

### Models
- Every model `copyWith` must support clearing nullable fields — use the sentinel pattern:
  ```dart
  static const _unset = Object();
  Audiobook copyWith({Object? title = _unset, ...}) =>
      Audiobook(title: title == _unset ? this.title : title as String?, ...);
  ```
- Models are immutable (`final` fields, `const` constructors where possible)

### Services
- Service classes must not be static-only utility bags over ~200 lines — split by format or responsibility (e.g. `Mp3Writer`, `Mp4Writer`, `FlacWriter`)
- Never hand-roll standard library functionality: use `dart:convert`'s `base64Encode` instead of a custom implementation

### Error Handling
- Never silently swallow exceptions with bare `catch (_) {}` in production paths — at minimum log to `debugPrint` or surface to the caller
- File I/O operations must propagate errors to the UI layer via a returned error list or `Result` type, not silent no-ops

### Testing
- Every public method in `lib/services/` with non-trivial logic must have a corresponding unit test in `test/services/`
- Every new model must have a `copyWith` round-trip test
- Widget tests must cover at least: initial render, a user interaction, and an error state

### Linting
- `analysis_options.yaml` must enable the additional rules listed in that file — do not remove them
- No `// ignore:` suppressions without an inline comment explaining why

## C++ (windows/runner/)
- Do not modify files under `windows/flutter/ephemeral/` — they are generated by the Flutter toolchain and will be overwritten on next build
- Scanner/review tools should exclude `windows/flutter/ephemeral/` from analysis
